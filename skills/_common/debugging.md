# Debugging Runbook

## 의사결정 트리

### IF 에러 메시지가 있음 (Debug)
1. 에러 메시지 전문 읽기 (마지막 줄만 보지 말고 전체 스택트레이스)
2. 가장 아래(root cause)부터 위로 읽기
3. **내 코드**에서 발생한 가장 깊은 프레임 찾기
4. 해당 라인의 입력값/상태 확인
5. 재현 조건 파악 → 최소 재현 코드 작성

### IF 에러 없이 잘못된 동작 (Debug)
1. 예상 동작 vs 실제 동작 명확히 정의
2. 입력 → 출력 경로를 추적 (데이터 흐름 따라가기)
3. 중간 지점에 로그/print 삽입하여 어디서 값이 달라지는지 확인
4. 이분 탐색: 경로의 중간 지점부터 확인하여 범위 좁히기

### IF 간헐적 발생 (Debug)
1. 재현 조건 수집 (시간대, 요청량, 특정 입력, 동시성)
2. 타이밍 관련이면 race condition 의심
3. 메모리 관련이면 누수/GC 의심
4. 로깅 강화하여 다음 발생 시 컨텍스트 수집

### IF 성능 문제 (Debug)
1. 어디가 느린지 측정 (추측하지 말고 프로파일링)
2. DB 쿼리: `EXPLAIN ANALYZE`로 실행 계획 확인
3. N+1 쿼리: ORM 쿼리 로깅 켜서 쿼리 수 확인
4. 메모리: 힙 스냅샷 비교 (시작 vs N분 후)
5. CPU: 프로파일러로 핫스팟 식별

### IF 환경 차이 (Debug)
1. "내 PC에서는 됨" → 환경 변수, 의존성 버전, OS 차이 확인
2. `.env` 파일 비교 (빠진 변수는 없는지)
3. `node_modules`/`venv` 삭제 후 재설치
4. Docker라면 이미지 재빌드 (`--no-cache`)

### IF 외부 시스템 통합 실패 (Debug)
외부 API/SDK/하드웨어/결제 단말 등 **통합 지점 실패** 시 외부 원인을 먼저 의심. 내 코드 조사는 후순위.

**적용 대상**: PG/VAN 연동, OAuth/SSO, 벤더 SDK, 카드 단말, IoT 장치, webhook 수신 등 제3자 의존 통합 전반.

**Triage 순서**:
1. **외부 시스템 단독 재현** — 벤더 앱/콘솔/REST 클라이언트로 직접 호출했을 때 같은 증상?
   - 동일 에러 → **외부 문제 확정**, 내 코드 조사 불필요
   - 내 앱에서만 재현 → 3단계로
2. **매핑/권한/설정 키워드** 확인 — 에러 메시지에 "매핑/상점/가맹점/권한/not found/unauthorized/invalid merchant" 등 → 벤더 측 등록 상태 이슈 확률 높음. 지원 연락
3. **연산 비대칭** — create는 되는데 update/cancel/delete만 실패? → 벤더의 리소스별 권한/매핑 비대칭이 일반적. 코드 아님
4. **그 다음에야 내 코드** — 요청 payload 덤프 → 필드 비교 → 타입/인코딩 검증

**시간 낭비 방지**: 1~3 스킵하고 바로 diff/로그 파기 시작하면 외부 원인을 내부로 오진단. 2-Strike로 반복 발생하는 패턴.

**프로젝트별 구체 사례**는 해당 도메인 스킬 참조 (예: `example_gateway-example_vendor.md`, `payment-pg.md`).

## 체계적 디버깅 프로세스

### 1단계: 재현
- 재현할 수 없으면 고칠 수 없음
- 최소 재현 조건 찾기 (불필요한 요소 하나씩 제거)
- 재현 스크립트나 테스트 케이스로 기록

### 2단계: 격리
- 문제 범위를 좁히기 (이분 탐색)
- 의존성 제거: 외부 서비스를 mock으로 대체하여 내부 문제인지 확인
- 최근 변경사항 확인: `git log`, `git bisect`

### 3단계: 진단
- 가설을 세우고 검증 (추측으로 코드 수정하지 말 것)
- 로그/디버거로 실제 상태 확인
- 한 번에 하나의 변수만 변경

### 4단계: 수정
- 근본 원인(root cause) 수정 (증상만 감추지 말 것)
- 수정 전 재현 테스트 작성 → 수정 후 통과 확인 (회귀 방지)
- 같은 패턴의 버그가 다른 곳에도 있는지 확인

## 실무 디버깅 워크플로우 (Git Flow + E2E)

> Playwright E2E 테스트에서 버그 발견 시 적용하는 실무 프로세스.
> Git Flow 브랜치 전략 기반. 단순 코드 수정이 아닌 **추적 가능한 이력**을 남김.

### IF E2E 테스트에서 버그 발견 (Playwright MCP)

```
1. 로그 탐색
   ├─ 브라우저 콘솔 로그 (browser_console_messages)
   ├─ 네트워크 요청/응답 (browser_network_requests)
   ├─ 백엔드 애플리케이션 로그 (docker logs)
   └─ DB 상태 확인 (실제 데이터 조회)

2. 이슈 생성
   ├─ 제목: [E2E] 재현 경로 요약
   ├─ 본문: 재현 단계 + 기대 동작 + 실제 동작 + 스크린샷/로그
   └─ 라벨: bug, 도메인 태그

3. Feature 브랜치 생성
   ├─ 브랜치명: fix/{이슈번호}-{짧은-설명}
   ├─ develop에서 분기 (Git Flow)
   └─ 재현 코드 분석 → 근본 원인 파악

4. 수정 + 검증
   ├─ 근본 원인 수정 (증상 우회 금지)
   ├─ 단위 테스트 추가 (회귀 방지)
   ├─ E2E 재실행으로 수정 확인
   └─ 관련 evaluator PASS 확인

5. PR 생성
   ├─ 제목: fix: {이슈 요약} (#{이슈번호})
   ├─ 본문: 원인 분석 + 수정 내용 + 테스트 계획
   └─ CI 전체 green 확인

6. 릴리즈 노트 기록
   ├─ CHANGELOG.md 또는 GitHub Release
   └─ [fix] 섹션에 사용자 관점 설명 추가
```

### 로그 탐색 상세 (Playwright MCP 환경)

| 로그 소스 | 도구/명령어 | 주로 확인하는 것 |
|----------|-----------|---------------|
| 브라우저 콘솔 | `browser_console_messages` | JS 에러, React 경고, 네트워크 에러 |
| 네트워크 | `browser_network_requests` | 4xx/5xx 응답, 요청 body 누락, CORS |
| 백엔드 로그 | `docker logs ecommerce-backend-1` | 스택트레이스, SQL 에러, Kafka 이벤트 |
| DB 상태 | `docker exec mysql ...` | 실제 저장된 데이터 vs 기대값 |
| Kafka | `docker exec kafka ... --topic` | 이벤트 발행 여부, 페이로드 내용 |

### 브랜치 전략 (Git Flow)

```
main ─────────────────────────────────── (프로덕션)
  └─ develop ────────────────────────── (통합)
       ├─ feature/bidding-system ────── (기능 개발)
       ├─ fix/42-deal-status-error ──── (버그 수정)
       └─ release/v1.1.0 ───────────── (릴리즈 준비)
```

- 기능 개발: `feature/{설명}` → develop PR
- 버그 수정: `fix/{이슈번호}-{설명}` → develop PR
- 핫픽스: `hotfix/{설명}` → main + develop PR
- E2E에서 발견한 버그는 **반드시 fix 브랜치**에서 수정 (develop 직접 커밋 금지)

## 자주 만나는 에러 패턴

### HTTP 상태 코드별 대응
| 코드 | 의미 | 첫 번째로 확인할 것 |
|------|------|---|
| 400 | 잘못된 요청 | 요청 body/params 형식, 필수 필드 누락 |
| 401 | 인증 실패 | 토큰 만료, 헤더 형식 (`Bearer ` 접두사) |
| 403 | 권한 없음 | CORS 설정, 리소스 소유자 확인 |
| 404 | 리소스 없음 | URL 오타, 라우트 등록 여부, trailing slash |
| 422 | 유효성 검증 실패 | DTO 필드 타입, enum 값 범위 |
| 429 | Rate Limit | Rate limiter 설정, 클라이언트 재시도 로직 |
| 500 | 서버 에러 | 서버 로그 확인, 처리되지 않은 예외 |
| 502/504 | 게이트웨이 | 업스트림 서비스 상태, 타임아웃 설정 |

### CORS 에러 디버깅
1. 브라우저 Network 탭에서 preflight OPTIONS 요청 확인
2. 서버 응답에 `Access-Control-Allow-Origin` 헤더 있는지 확인
3. 프론트/백엔드 포트가 다르면 CORS 설정 필수
4. credentials(쿠키) 사용 시 와일드카드(`*`) 불가 → 명시적 오리진

### DB 관련 디버깅
- 연결 실패: 호스트/포트/인증 정보, 방화벽, max_connections 초과
- 느린 쿼리: `EXPLAIN ANALYZE`, 인덱스 누락, 풀 테이블 스캔
- 데드락: 트랜잭션 순서 불일치, 락 타임아웃 설정
- 마이그레이션 실패: 스키마 불일치, 데이터 마이그레이션 누락

## Gotchas

### 증상만 고치기
`try/except: pass`로 에러를 삼키거나, 조건문으로 특수 케이스만 우회하는 것은 수정이 아님. 근본 원인을 찾아 고칠 것.

### 추측으로 코드 변경
"아마 이게 문제일 거야"로 코드를 수정하면 원래 문제를 감추고 새 문제를 만듦. 반드시 가설 → 검증 → 수정 순서.

### 로그 없이 디버깅
print/console.log를 임시로 넣는 것은 괜찮지만, 구조화된 로깅(structlog, winston)을 프로덕션에 설치해두는 것이 장기적으로 훨씬 효율적.

### git bisect 미사용
"언제부터 안 됐지?" → `git bisect`가 이분 탐색으로 문제 커밋을 찾아줌. 수동으로 커밋을 하나씩 체크아웃하지 말 것.

### Windows 줄바꿈 문제
Windows에서 만든 파일을 Linux에서 실행하면 `\r` 때문에 에러. 에러 메시지에 `\r`, `^M`, `bad interpreter`가 보이면 줄바꿈 문제.

### 환경변수 미로드
`.env` 파일이 있어도 dotenv 라이브러리를 로드하지 않으면 적용 안됨. 또한 `.env` 변경 후 서버 재시작 필요 (자동 반영 아님).

### async/await 누락
async 함수 호출 시 `await`를 빠뜨리면 Promise 객체가 반환되어 조건문이 항상 truthy. 에러도 안 나고 잘못된 동작만 발생하므로 찾기 어려움.

### Docker 내부 localhost
컨테이너에서 `localhost`는 컨테이너 자신을 가리킴. 다른 컨테이너의 서비스에 접근하려면 서비스명(docker-compose) 또는 호스트 IP 사용.

## 워크플로우 사후 분석 (Forensics — GSD 흡수)

파이프라인이나 작업 흐름이 실패했을 때 "왜 실패했는가"를 체계적으로 분석하는 절차.
하네스 2원칙(자기 개선 루프)의 핵심 도구.

### 분석 절차
1. **상태 수집**: `git log --oneline -20`, `git status`, `git diff --stat`
2. **타임라인 재구성**: 커밋 시간순으로 무엇이 언제 변경되었는지 정리
3. **이상 탐지**: 비정상 패턴 식별
   - 같은 파일 반복 수정 (3회 이상 → 설계 문제)
   - 커밋 간 시간 격차 (30분+ → 막힌 지점)
   - revert/amend 커밋 (시행착오)
4. **근본 원인 분류**:
   - 설계 오류 → Designer 단계 부족 (DGE 1원칙 위반)
   - 구현 오류 → Generator 산출물 불량
   - 검증 누락 → Evaluator 단계 건너뜀
   - 환경 이슈 → 인프라/설정 문제
5. **교훈 추출 → 스킬/훅 반영** (2-Strike Rule과 연계)

### 사후 분석 트리거
- 같은 에러를 3회 이상 수정한 경우
- 파이프라인 단계가 예상 시간의 2배 이상 걸린 경우
- 사용자가 "왜 이렇게 됐지" 등 원인 분석을 요청한 경우
